What is your immediate diagnosis?

Diagnosis Logged. The answer: The two tables were never formally linked! They were just two islands of data. Today, we build the bridges and enforce the ultimate law: Referential Integrity.

📚 The City Planner's Cheat Sheet

Select your difficulty level to explore the core concepts of relational databases.

Primary & Foreign Keys

Primary Key: A field that uniquely identifies each record in its own table; cannot be null or duplicated.
Analogy: The citizen's biometric chip ID — unique to one person, forever.

Foreign Key: A field in one table that references the primary key of another table, creating a link between them.
Analogy: The CitizenID field printed inside an ENERGY_BILL — it points back to the real person in the CITIZENS registry.

Flat-file vs Relational: A flat file is one giant spreadsheet causing redundancy. A relational database splits data across linked tables.

Relationship Types

A relationship is a defined link between two tables based on matching key fields.

Type Description Medellín Analogy
One-to-One One record matches exactly one record One citizen ↔ one biometric profile
One-to-Many One record matches many records One district ↔ many citizens (This is the standard!)
Many-to-Many Many records match many records Many citizens ↔ many transport routes

⚠️ Exam Trap: Never say "two-to-many" or "multiple-to-one". Use strictly one-to-one, one-to-many, or many-to-many.

Referential Integrity & Cascades

Referential Integrity: A rule that prevents a foreign key value from existing without a matching primary key value in the related table.
The City Law: "You cannot bill a citizen who doesn't exist."

Cascade Options

  • Cascade Update: If a Primary Key value changes, all matching Foreign Key values update automatically. (Safe to use).
  • Cascade Delete: If a record is deleted from Table A, all linked records in Table B are deleted automatically. (Dangerous! Deleting a district could delete all its citizens).

🛠️ The Live Build: Wiring the Connections

Follow along in MS Access (`Medellin2050_SmartCity.accdb`) as we connect the tables.

Step 1: The DISTRICTS Reference Table

Create a new table in Design View saved as DISTRICTS. Set DistrictCode as the Primary Key.

Why do we do this first? You must build the reference table (the parent) before you can link anything to it.

Step 2: Add Foreign Key to CITIZENS

Open CITIZENS in Design view. Add a new field: DistrictCode (Short Text, Size 5).

Notice: This field has the exact same name and data type as the Primary Key in DISTRICTS. This is required!

Step 3: Enforce the Relationship (The Tests)

In Database Tools → Relationships, drag from the Primary Key to the Foreign Key. Tick "Enforce Referential Integrity".

What happens if you try to add BillID 13 to CitizenID "C-00999"?
Access blocks it! "You cannot add a record because a related record is required in CITIZENS." The city's law is working.
What happens if you try to delete Citizen C-00101?
Access blocks it! "The record cannot be deleted because ENERGY_BILLS includes related records." You cannot demolish a citizen while bills are attached!

🔴 Stretch Challenge

Line-D is decommissioned. The city engineer wants to delete its record from METRO_LINES. How do you do this without violating referential integrity?

First, update or delete all child records in METRO_VEHICLES where LineCode = 'Line-D'. Once no child records reference the foreign key, you can safely delete the parent record from METRO_LINES.

📖 Smart City Glossary

Primary Key
A field that uniquely identifies each record in its own table; cannot be null or duplicated. (The citizen's biometric ID).
Foreign Key
A field in one table that references the primary key of another table, creating a link between them. (The DistrictCode inside the CITIZENS table).
Relationship
A defined link between two tables based on matching key fields. (The bridge between two islands of data).
Referential Integrity
A rule that prevents a foreign key value from existing without a matching primary key value in the related table. (You cannot bill a ghost).
One-to-Many
One record in Table A can be linked to many records in Table B. (One district, many citizens).
Flat-file
All data stored in a single table, causing massive redundancy.
Cascade Update
If a PK value changes, all matching FK values update automatically.
Cascade Delete
If a parent record is deleted, all linked child records are deleted automatically. (Use with caution!).
