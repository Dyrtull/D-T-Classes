Concept Exploration

2A — Calculated Fields: The Exact Exam Patterns

A calculated field is a column that does not exist in any table. It is computed from other fields at the moment the query runs. It is never stored — it is always recalculated fresh each time the query executes.

The syntax is always the same — type this into the Field row of a blank QBE column:

AliasName: [Field1]*[Field2]

The alias is the column heading that will appear in the report. The square brackets contain the exact field names from the table. The operator can be *, /, +, or -.

Example: AnnualBonus: [Salary]*0.1
This creates a new column called AnnualBonus, showing 10% of each record's Salary value.

Every calculated field from real Cambridge past papers — the pattern is always one or two arithmetic operations:

PaperAliasFormulaWhat it calculates
FM2021OrderValue[NetPrice]*[ReorderNo]Value of restocking one product
FM2021OrderNow[Available]<=[MinStock]Boolean: does this need reordering?
FM2024Sales[GlobalSales]/354*100Percentage of total market sales
Session 3AnnualBonus[Salary]*0.110% of salary
Formatting the calculated field: After typing the formula, right-click the column header in the QBE grid → Properties → set Format and Decimal Places. The question always specifies the format. A calculated field showing 450000.0000 instead of £450,000.00 loses the formatting mark.
The #Error trap: If your calculated field shows #Error in the query or report, the field name in square brackets does not exactly match the actual field name in the table — including spaces, hyphens, and capitalisation. [GlobalSales] is not the same as [Global Sales]. Open the table in Design View, check the exact field name character by character, and correct the formula.

The calculated field exists only in the query output — it is not stored back into the table. Each time you run the query, Access reads the source fields and computes the value fresh. This means:

  • If you update Salary in the table, the AnnualBonus value in the query changes automatically on next run
  • You cannot filter the calculated field in the same query where it is created — you must use a second query if you need to filter on it
  • The calculated field does not appear in the table's Design View — it only exists in the query

Vocabulary Reference

Click any term to reveal its definition. Use the search box to filter.

Calculated Field
A field in a query that does not exist in any table — it is computed at run-time from other fields using an arithmetic formula. Syntax: AliasName: [Field1]*[Field2]. Typed in the Field row of a blank QBE column. Never stored back to the table — recalculated fresh each time the query runs. Appears in the query output and report as a normal column.
Alias (Calculated Field)
The name given to a calculated field — the heading that appears in the query output and report. Placed to the left of the colon in the QBE Field row. For example, in "AnnualBonus: [Salary]*0.1", AnnualBonus is the alias. The alias is given by the exam question — it must be used exactly as specified.
OR Criteria (QBE)
In the Access QBE grid, OR conditions are placed on separate rows — the Criteria row for the first condition and the Or: row for the second. Multiple criteria on the same row are joined by AND; criteria on different rows are joined by OR. The critical rule: AND lives on the same row; OR lives on a different row.
Mixed AND/OR Criteria
A query pattern where some conditions are AND (same row) and others are OR (different rows). For "GameTitle includes Tomas AND AgeCode is C3 or C7": the Criteria row has Like "*Tomas*" + "C3"; the Or: row has Like "*Tomas*" + "C7". The wildcard must be repeated on both rows because each row is an independent condition set — omitting it from the Or: row returns all C7 records regardless of title.
Independent Condition Set
Each row in the QBE Criteria area is treated as a complete, independent condition. A record is selected if it satisfies ANY row in its entirety. Criteria within the same row are all required simultaneously (AND). Empty cells in a row impose no restriction on that field for that condition. This is why the wildcard must be repeated on every row where the title restriction applies.
=Avg([FieldName])
An Access aggregate formula that calculates the arithmetic mean of all values in the specified field across all records in the report. Placed in the Report Footer section, aligned directly below the column it summarises. The label goes to its left. Formatted as Currency (2dp) for money fields. The formula text — not the calculated result — must be fully visible in the Design View evidence screenshot.
Report Footer
The section at the very end of a report that appears once after all data rows. Used for grand total aggregate formulas — =Count(*), =Sum([Field]), =Avg([Field]). Fires once per report run. Not to be confused with Page Footer (repeats on every page) or Group Footer (fires once per group). For all ungrouped reports in Sessions 2 and 3, all aggregates go in the Report Footer.
Group Footer
A report section that appears after the records belonging to each group, when grouping is applied. Its formula calculates only across the records in that specific group. For a report grouped by DeptName with =Sum([Salary]) in the Group Footer, each department would show its own salary subtotal. Not covered until Session 4. For ungrouped reports: use Report Footer.
#Error (Calculated Field)
An error value displayed in every row of a calculated field when the formula cannot execute. The most common cause is a field name mismatch — the name inside square brackets does not exactly match the actual field name in the table, including capitalisation and spacing. Fix: open the table in Design View, verify the exact field name, and correct the formula. #Error on all rows = name mismatch. #Error on some rows = possible division by zero or null value.
30-Second Parse Method
A systematic technique for decoding a Report 2 exam question before touching Access. Five elements in order: (1) Tables and join, (2) Criteria — wildcard, Boolean, OR rows needed, (3) Calculated field — alias and formula, (4) Aggregate — formula, column, label, (5) Format — orientation and candidate details location. Applied before building the query, this method prevents errors caused by building without fully reading the question.
Aggregate Formula (Alignment)
The requirement to position an aggregate formula textbox directly below the column it summarises, not floating in the centre of the Report Footer. In Design View, drag the textbox left or right until it is horizontally aligned with the column header above. The mark scheme checks position — a correctly calculated formula in the wrong position can still lose the placement mark.
Run-Time Calculation
A calculation performed at the moment the query executes, rather than stored in the table. Calculated fields in the QBE grid are run-time calculations — they use the current values of source fields each time the query runs. If a source field's value changes, the calculated field result changes automatically on the next run. This is why calculated fields cannot be filtered within the same query that creates them — the value does not exist until the query runs.
