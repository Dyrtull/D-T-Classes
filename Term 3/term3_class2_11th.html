The Hook: Table vs Report

The city's procurement director receives your database. She opens it. She sees 10 rows of raw data in a table — fields everywhere, no order, no totals, no titles. She closes it and calls IT support.

That is a zero. Not because your data was wrong. Because you presented it as a table instead of a report.

The one-sentence distinction: A table is where data lives. A report is what the examiner marks. From this session forward, every piece of work submitted from a database is a formatted, printed report — with a title, a sort, a total at the bottom, and your candidate details in the right place.

Output B: A printed report with a title in larger font, a =Sum formula below the correct column, candidate details in the footer, and portrait orientation — all records and sorts matching Output A.

Concept Exploration

2A — The Query-to-Report Pipeline

The report is never built directly from a table. Every Paper 2 report follows this exact four-stage pipeline:

TABLE(s)Where data lives — never submitted directly
↓
QUERYCriteria + sort + calculated fields
↓
REPORTFormatting + totals + headers/footers
↓
PRINTOUT / PDFWhat the examiner marks
The most common exam mistake: Students add criteria directly in the Report Wizard and skip building the underlying query. This works for very simple single-field criteria — but fails immediately for wildcard patterns (Like "*bean*"), OR criteria across two rows, and calculated fields. Always build the query first, every time, without exception.

The query and the report each have a separate job:

StageJobWhere you work
QuerySelect correct records, apply criteria, set sort orderQuery Design View (QBE grid)
ReportFormat the output, add title, add totals, add candidate detailsReport Design View / Wizard

There are two orderings to keep mentally separate at all times:

QBE column order — controls sort priority. The leftmost sorted field in the QBE grid is the primary sort. This is independent of which column is displayed first in the report.

Report column order — the left-to-right sequence of fields in the report output. Controlled by the order you add fields in the Report Wizard, or by dragging controls in Design View. The exam question always says "in this order" — this order applies to the report, not the QBE grid.

These are two separate orderings serving two different purposes. Conflating them is the source of almost every sort-order error in the exam.

2B — Wildcard Criteria: The Exact Exam Syntax

An asterisk (*) in an Access criterion means "any sequence of characters". Three positions — three meanings:

CriterionMatchesDoes NOT match
Like "*bean*"Runner Bean, French Bean, BeansproutTomato, Lettuce
Like "bean*"Beansprout, BeanRunner Bean (doesn't START with bean)
Like "*bean"Runner Bean, Soy BeanBeansprout (doesn't END with bean)

The exam almost always uses Like "*word*" — contains the word anywhere. Memorise this pattern.

Exact exam phrasings and the criterion to type:

Exam wordingCriterion to type
"includes the word bean"Like "*bean*"
"Type includes the word bean"Like "*bean*" (in the Type row)
"begins with the text Cyrus"Like "Cyrus*"
"contains Tomas"Like "*Tomas*"
"ServiceGrade includes the word Senior"Like "*Senior*"
Case sensitivity: Access wildcards are NOT case-sensitive. Like "*bean*" matches "Bean", "BEAN", "bean" — all correct. You do not need to type the criterion in lowercase.
The no-asterisk trap: Like "bean" (no asterisks) matches ONLY records where the entire field value is exactly the word "bean" — no more, no fewer characters. Since the Type field contains values like "Runner Bean", not just "bean", Like "bean" returns zero records. Every wildcard criterion in the exam uses at least one asterisk.

Why this matters for the mark scheme: criteria are awarded a mark when the correct records appear in the report output. If your wildcard is wrong, the wrong records appear, and zero criteria marks are awarded — even if everything else (sort, totals, formatting) is perfect.

2C — Double Sort: The Mark-Scheme Rule

When the exam specifies two sort fields, the primary sort is applied first. Within records that share the same primary sort value, the secondary sort orders those tied records.

Example: "Descending order of SeedID, then ascending order of DatePlaced."
— Primary: SeedID descending (SD-010 first, SD-001 last)
— Secondary: within any records with the same SeedID, DatePlaced ascending (earliest order first)

The critical rule: In the Access QBE grid, the leftmost sorted field is the primary sort.

For the report shown in the demo (Salary descending primary, Surname ascending secondary), the QBE grid must look like this:

Salary (primary sort) must appear to the LEFT of Surname (secondary sort) in the QBE grid. Even though in the final report Surname will appear as the first column, in the QBE grid sort priority is determined by horizontal position.

The column-order trap: If a student adds fields to the QBE grid in the order the report requires (Surname, FirstName, DeptName, ServiceGrade, Salary) and then sets sorts on Salary and Surname, Surname is now to the LEFT of Salary — so Surname becomes the primary sort, not Salary. The report will sort alphabetically by surname instead of by salary. To fix: drag the Salary column left of Surname in the QBE grid. This does not change the report column order — it only changes sort priority.

2D — Report Totals: The Three Exam Formulas

Report 1 always asks for exactly one aggregate formula placed at the bottom. Only three formulas appear across all past papers:

FormulaWhat it calculatesWhen to use it
=Count(*)Total number of records"total number of [things]"
=Sum([FieldName])Total of all values in that field"total value" / "total number of packets ordered"
=Avg([FieldName])Average value of that field"average price" / "mean salary"

Placement rules — always the same, every paper:

1

The formula goes in the Report Footer section — not Page Footer, not Detail, not Report Header.

2

A label goes immediately to the LEFT of the formula textbox — not above, not below, not right.

3

The label text must match the question specification character by character. "Total salary cost:" is not the same as "Total Salary Cost" or "Total salary cost" (without the colon).

4

The formula value must be formatted correctly — a count formatted as 40.00 loses the mark. Set Decimal Places to 0 for counts, Currency format for money totals.

Count(*) vs Sum([Packets]) — the most common formula error: If 4 orders have 10, 25, 20, and 12 packets respectively:
=Count(*) = 4 (there are four records)
=Sum([Packets]) = 67 (the total of all packet values)

The question "calculates the total number of packets ordered" means the total quantity — 67. Use =Sum([Packets]). The question "calculates the total number of orders" means the record count — 4. Use =Count(*). Read the wording carefully every time.
The Evidence Screenshot requirement: After placing the formula, the exam always asks for a screenshot taken in Design View showing the formula text — not the calculated result, the actual formula string. Widen the textbox first. If the textbox is too narrow and shows =Sum([Net... instead of =Sum([NetPrice]), the mark is lost. This is a standalone mark on every single paper.

Q1 (2 marks)

Explain the difference between a query and a report in a database application, and state which one is printed for the examiner.

Mark Scheme:
A query extracts and filters records from tables based on specified criteria, returning a raw datasheet of matching records (1 mark). A report takes data from a query or table and presents it in a formatted, printable layout with titles, sorting, totals, and custom headers and footers — this is what is printed for the examiner (1 mark).

Vocabulary Reference

Click any term to reveal its definition. Use the search box to filter.

Query
A database object that extracts records from one or more tables based on specified criteria, sort orders, and field selections. In the exam pipeline, the query is always built before the report. It acts as a filtered, sorted view of the underlying data. Saved with a descriptive name (e.g. Q_Report1_SeniorStaff).
Report
A database object that presents data from a query or table in a formatted, printable layout. Includes a title, column headers, data rows, aggregate totals in the footer, and candidate details. The report — not the query, not the table — is what the examiner marks. Always submitted as a printed page or PDF.
QBE Grid (Query By Example)
The visual grid in Access Query Design View where fields, sort directions, and criteria are specified. Each column in the QBE grid represents one field. Sort priority is determined by horizontal position — the leftmost sorted column is the primary sort. Criteria on the same row are joined by AND; criteria on different rows are joined by OR.
Wildcard Criterion
An Access criterion using the asterisk (*) as a placeholder for any sequence of characters. Syntax: Like "pattern". Like "*word*" matches any value containing "word" anywhere. Like "word*" matches values starting with "word". Like "*word" matches values ending with "word". Access wildcards are case-insensitive.
Primary Sort
The first sort applied to records — the field by which records are ordered first. All records are arranged according to this field first. The secondary sort only applies to groups of records that share the same primary sort value. In the Access QBE grid, the primary sort is the leftmost column with a sort direction set.
Secondary Sort
The sort applied within groups of records that share the same primary sort value. For example, if the primary sort is Salary descending, records with the same salary value are then ordered by the secondary sort (e.g. Surname ascending). In the QBE grid, the secondary sort field must be to the RIGHT of the primary sort field.
Report Header
The section at the very top of the report that appears once, before any data rows. Contains the report title. The title must be in a larger font size than the body text — this is a dedicated mark on the exam. Does not repeat on subsequent pages.
Report Footer
The section at the very end of the report that appears once, after all data rows. This is where aggregate formula totals are placed — =Count(*), =Sum([Field]), =Avg([Field]). The formula label goes to the LEFT of the formula textbox in this section. Not to be confused with Page Footer.
Page Header
The section that appears at the top of every page of the report, above the data rows. Contains column headings by default. The exam may specify that candidate details go in the Page Header (e.g. Report 2 in FM2025).
Page Footer
The section that appears at the bottom of every page. By default, Access auto-generates page numbers and dates here. These must be DELETED before adding candidate details — auto-generated content is not candidate details and will not earn the candidate details mark. Delete everything first, then add the candidate name, centre number, and candidate number.
=Count(*)
An Access aggregate formula that counts the total number of records in the report. Returns the number of rows, not the sum of any field value. Used when the exam asks for "total number of [things]" where the things are records. Formatted as Integer (no decimal places) — a result of 8.00 instead of 8 loses the formatting mark.
=Sum([FieldName])
An Access aggregate formula that adds up all values in the specified field across all records in the report. Used when the exam asks for "total value" or "total number of packets ordered" (total quantity, not total records). The field name must be enclosed in square brackets. Formatted as Currency for money totals, Integer for quantity totals.
=Avg([FieldName])
An Access aggregate formula that calculates the arithmetic mean of all values in the specified field across all records in the report. Used when the exam asks for "average price" or "mean salary". Formatted as Currency for money averages. Appears in Session 3's Report 2 pattern.
Tabular Layout
A report layout where each record occupies one row, with fields displayed in columns across the page — similar to a spreadsheet or table. The standard layout for all Cambridge Paper 2 reports. Selected in the Report Wizard layout screen. Alternative layouts (Columnar, Justified) are not used in this exam.
Evidence Document Screenshot
A required screenshot taken in Access Design View showing the formula in the Report Footer textbox. The formula text (e.g. =Sum([Salary])) must be fully visible — not the calculated result. Taken after widening the textbox to ensure no truncation. A standalone mark on every Cambridge Paper 2 database task.
PDF Export
The process of saving a report as a PDF file using External Data → PDF or XPS. The filename is specified in the exam question (usually uppercase). The file extension .pdf must be visible in the folder screenshot. If Windows is hiding extensions, enable them via View → Show → File name extensions before taking the screenshot.
