Don't Be the Mars Orbiter

Grade 11 — Cambridge IGCSE ICT | Database Session 1 of 11

The Hook: A $327 Million Question

23 September 1999. NASA's Mars Climate Orbiter has been travelling for 9 months and 416 million miles. It is 60 seconds from entering Mars orbit. Then it disappears. $327 million — gone.

The investigation identified the cause in three sentences: one engineering team was transmitting navigation data in imperial units — pound-force seconds. The software receiving that data expected metric units — newton-seconds. Nobody had checked. The spacecraft received perfectly valid numbers. The numbers just meant the wrong thing. The Orbiter flew 170 kilometres too close to Mars and burned up in the atmosphere.

The crucial detail: The numbers were not wrong. They were stored correctly. They were transmitted correctly. The catastrophic error was in the type — not the value itself.

Concept Exploration

2A — The Six Data Types the Exam Always Uses

From analysis of every Cambridge Paper 2 past paper, only these six data types appear in the database task. You need to know all six — and the difference between Store and Display for each.

The six data types are like different units of measurement. Using the wrong one is like writing a speed in kilometres but the car's computer expects miles per hour — the number looks fine, the outcome is catastrophic.

The Mars Orbiter scenario applies directly to every import. The danger is not wrong values — it is correct values with wrong types. Three cases to memorise:

'18.80' as Short Text — looks identical to 18.80 (Currency) in Datasheet View. But: SUM([Price]) returns 0 or an error. SORT([Price]) sorts alphabetically (2.45 before 18.80). The wrongness is invisible until the query runs. '15/03/2040' as Short Text — looks identical to 15/03/2040 (Date/Time) in Datasheet View. But: You cannot filter dates with > or < operators. DateDiff() returns an error. 'Yes' as Short Text — looks identical to Yes (Boolean) in Datasheet View. But: A criterion of = Yes (Boolean) returns no results. The text string "Yes" and the Boolean Yes are different types entirely.
Why this matters for the mark scheme: The examiner cannot see your Datasheet View values. They open your database in Design View. If IsActive is Short Text storing "Yes"/"No", you receive 0 marks for that field's data type — even though the values look identical on screen to Yes/No Boolean.

2B — Primary Keys: The One Rule

Exam-Precision Definition: A primary key is a field whose value uniquely identifies each record in a table. No two records can share the same primary key value, and it cannot be left empty.

Think of it as a national identity number — every person has one, no two people share one, and it cannot be blank.

How to identify the primary key when the exam says "examine the file and identify the most appropriate field" — apply this test in order:

1

Is there a field with Code, ID, or Ref in the name? Almost certainly the primary key.

2

Does every record have a unique value in that field? Scan the CSV — confirm no duplicates.

3

Could it ever be blank? If yes, it cannot be the primary key.

Past paper primary keys: ProductCode, GameCode, SeedID, AgeCode, GenCode.

The AutoNumber Trap — loses marks every time: When Access imports a CSV, it may offer to create an AutoNumber primary key. Always choose "Let me choose my own primary key" and select the correct field. An AutoNumber PK on an imported table is almost always wrong — it has no relationship to the actual data, so it cannot be used to join tables or referenced in reports.

Why does the exam use codes like "SD-001" rather than simple integers? A code field communicates meaning: SD tells you it is a Seed record, 001 gives sequence. An AutoNumber integer communicates nothing. The examiner wants you to understand that primary keys are chosen for semantic reasons, not just uniqueness.

2C — Relationships: The Only Pattern the Exam Uses

Every Paper 2 past paper uses the same relationship structure: one parent table, one child table, linked by a shared field.

TABLE A (one side) TABLE B (many side) Primary Key: FieldX → Foreign Key: FieldX 1 ∞

Example: One department can have many officials. DEPARTMENTS is the "one" side; OFFICIALS is the "many" side.

Three things the mark scheme checks — all three must be visible in your screenshot:

1

The relationship line exists between the correct fields (1 mark)

2

The 1 and infinity symbols are visible — this confirms referential integrity was enforced

3

Only one screenshot is needed — the Relationships window showing both symbols

Real examples from past papers: SupplierCode in suppliers → SupplierCode in stores; SeedID in seeds → Seed in orders; GenCode in genre → GenCode in games.

The Field Name Trap: In the Seeds paper, the primary key field is SeedID but the foreign key in orders is called Seed — different names, same data, valid join. Access will NOT draw the line automatically — you must drag manually from SeedID in the seeds table and drop it onto Seed in the orders table.

Referential integrity settings for the exam:

  • Enforce Referential Integrity — always tick this
  • Cascade Update Related Fields — tick this
  • Cascade Delete Related Records — never tick this in an exam context

2E — Adding a New Record: The Referential Integrity Test

In every Paper 2 paper, Step 4 asks you to add a new record to one of your tables. This is worth 1–2 marks.

Before entering the record, ask: Does every foreign key value in the new record already exist in the parent table?

The exam uses two variants of Step 4:

A

Straightforward: The foreign key value in the new record already exists in the parent table. Enter the record directly.

B

The trap: The foreign key value does not exist in the parent table yet. You must add it to the parent table first, then add the child record. Attempting to add the child record first causes a referential integrity error — Access will reject it.

The error message is not failure — it is the database working correctly. When Access rejects a record with a referential integrity error, it is protecting the database from orphan records — child records that reference a parent that does not exist. This is exactly the kind of silent error that crashes spacecraft. The error message is informative. Read it, identify which foreign key is the problem, fix the parent table, then re-enter the record.

Vocabulary Reference

Click any term to reveal its definition. Use the search box to filter.

Short Text
An Access data type that stores up to 255 characters of text — letters, numbers, symbols, or any combination. Used for codes, names, and any field that does not need arithmetic. The exam instruction will simply say "Text".
Number (Integer)
An Access data type that stores whole numbers only — no decimal places. Set the Field Size to Integer or Long Integer in the field properties. Exam instruction: "Integer" or "0 decimal places". Used for fields like StockLevel, Packets, ReleaseYear.
Currency
An Access data type that stores monetary values with fixed decimal precision. Always displays with a currency symbol and 2 decimal places when the Format property is set to Currency. Exam instruction: "currency symbol, 2 decimal places". The import wizard often guesses this field as Long Integer — correct it in Design View.
Date/Time
An Access data type that stores dates and times in a way that supports correct sorting, filtering, and calculations. The Format property controls display (e.g. dd/mm/yyyy or dd-MMM-yy). Exam instruction will specify the exact display format. The import wizard will set DateOrder — always verify it is DMY for UK-format dates.
Yes/No (Boolean)
An Access data type that stores exactly two values: Yes or No (internally stored as -1 and 0). The import wizard almost always sets Boolean-looking fields as Short Text. This must be corrected in Design View. A Short Text field storing "Yes"/"No" and a Yes/No Boolean field look identical in Datasheet View but behave entirely differently in queries.
Primary Key
A field whose value uniquely identifies each record in a table. No two records can share the same primary key value, and it cannot be left empty (null). Identified in the exam by looking for a field with Code, ID, or Ref in the name, verifying uniqueness, and confirming it cannot be blank. Never use an AutoNumber field as a primary key for an imported table.
Foreign Key
A field in a child table that references the primary key of a parent table. The foreign key and primary key fields contain matching values that link records across tables. In the exam, the foreign key field may have a different name from the primary key field it references — in that case the relationship line must be drawn manually.
One-to-Many Relationship
A relationship type where one record in the parent table can correspond to many records in the child table. In the Relationships window, the "1" symbol appears on the parent (primary key) side and the infinity symbol appears on the child (foreign key) side. Every Paper 2 past paper uses this relationship type.
Referential Integrity
A database rule that ensures every foreign key value in the child table matches an existing primary key value in the parent table. When enforced, Access prevents adding child records with invalid foreign key values and displays an error if you attempt to do so. This error is correct behaviour — the database is protecting itself from orphan records.
Orphan Record
A child record whose foreign key value does not match any primary key value in the parent table. For example, an order record referencing SeedID = SD-099 when SD-099 does not exist in the seeds table. Referential integrity prevents orphan records from being created.
Design View
The Access view that shows the structure of a table — field names, data types, and properties — rather than the data itself. Every post-import audit must be performed in Design View. The import wizard's guesses are visible here and can be corrected before the database is used.
Datasheet View
The Access view that shows the actual data stored in a table in rows and columns. This view does not show data types — a Short Text "Yes" and a Boolean Yes look identical here. This is why wrong data types are silent errors: they are invisible in Datasheet View but cause failures in queries and calculations.
Import Wizard
The step-by-step dialog in Access that guides you through importing a CSV file as a new table. The wizard makes guesses about data types based on the first few rows of data. These guesses are often wrong — especially for Boolean fields (guessed as Text) and Currency fields (guessed as Long Integer). The Design View audit corrects the wizard's mistakes.
AutoNumber
An Access data type that automatically assigns a unique sequential integer to each new record. The import wizard may offer to create an AutoNumber primary key — always decline this for imported tables and choose your own primary key instead. An AutoNumber field has no relationship to the actual data and is almost always wrong as a primary key in an exam context.
Field Size Property
A property of Short Text and Number fields in Design View. For Short Text, it sets the maximum number of characters (default 255, should be reduced to match the actual data). For Number, it determines the subtype: Integer (whole numbers, –32,768 to 32,767), Long Integer (larger whole numbers), Single/Double (decimal). Always reduce Short Text sizes from 255 to something sensible after import.
Null
The absence of a value in a field — different from zero, from an empty string, or from the word "blank". A Date/Time field can validly be null if no date has been recorded yet (for example, an order that has not been completed has no DateCompleted). A null is not an error.
