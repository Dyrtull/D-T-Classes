Basic Level: The Essentials

The Eight Data Types (Simplified)

Think of data types as containers. Each container only holds one kind of thing:

📝 Text (Short Text)

Stores letters, numbers, and symbols combined. Max ~255 characters.

Examples: CitizenID (C-00451), District Name, Status

🔢 Number (Integer)

Stores whole numbers only. Allows math operations.

Examples: Age (25), Capacity (120), Population

💵 Currency

Stores money values with automatic formatting and rounding.

Examples: MaintenanceCost ($5,200), Fare

📅 Date/Time

Stores dates and times so they can be sorted chronologically.

Examples: CommissionDate (25/03/2050), LastServiceDate

☑️ Yes/No (Boolean)

Stores only two values: True or False (displayed as checkboxes).

Examples: IsElectric, IsVerified, IsActive

🔑 AutoNumber

Automatically generates unique numbers (1, 2, 3...). Cannot be edited.

Examples: Auto-generated ID fields

📦 OLE Object & Memo

Memo: Long text (up to 65,000 characters)

OLE Object: Embedded files (images, documents)

Examples: Notes, Description, Photos

Intermediate Level: Design Decisions

Why Data Types Matter

Data types aren't just labels — they're contracts. When you declare a field as Currency, you're telling the database: "I promise this will only contain money values, and you should format it with a currency symbol and two decimal places."

🔴 Why the Metro Crash Happened →

The Database Knew How to Store Text. But the AI couldn't sort "thirty-two" numerically. It couldn't calculate "next Tuesday" as a future date. It couldn't charge a fare in currency format.

The System Failed because the schema didn't enforce the business rules. The data types were wrong.

The Primary Key: The City's Fingerprint

Definition (IGCSE): A primary key is a field (or combination of fields) that uniquely identifies each record. It cannot be null and cannot contain duplicates.

🔑 Properties of a Primary Key:
  • Unique: No two records can have the same value
  • Never empty (non-null): Every record must have a value
  • Stable: The value should not need to change
  • Minimal: As short as possible while remaining unique

Analogy: Every citizen of Medellín 2050 has a chip implanted at birth with a unique 7-character ID. No two chips are identical—even twins have different IDs. That chip ID is the primary key of the CITIZENS table.

Flat-File vs. Relational Databases

📋 Flat-File Database →

Definition: All data stored in a single table.

Problem: Data redundancy. If Medellín has 10 citizens in District Centro, the district information is repeated 10 times.

Risk: Update anomaly—changing the district manager requires updating 10 rows, and you might miss one, creating inconsistency.

🔗 Relational Database →

Definition: Multiple tables linked by keys. Each data element stored only once.

Advantage: Change the district manager once, and all citizens instantly see the update. No redundancy. No inconsistency.

Best for: Complex systems like cities where relationships matter (citizens → districts → energy grids → metro stations).

Advanced Level: Schema Design & Optimization

The Data Type Selection Algorithm

When designing a schema, ask these questions in order:

1. Is this a unique identifier? → AutoNumber or a meaningful Text field (e.g., CitizenID)

2. Is this currency? → Currency (not Number). Currency automatically handles rounding and formatting per local standards.

3. Is this a date or time? → Date/Time. Never store as Text. Enables chronological sorting, calculations (age, duration), and query filtering by date range.

4. Is this a yes/no value? → Yes/No (Boolean). Prevents invalid entries like "YES", "yes", "y", "1" cluttering the database.

5. Is this numeric and mathematical? → Number (choose Integer or Double based on decimal need). Allows SUM, AVG, and comparison operators.

6. Is this very long (>255 chars)? → Memo. For notes, descriptions, comments.

7. Otherwise: → Short Text. Default for names, addresses, codes.

Natural Keys vs. Surrogate Keys

Natural Key (Composite)

A key based on real-world data: CitizenID (C-00451), VehicleCode (MV-A001)

Pros: Meaningful, self-describing, users understand it

Cons: Can change (city might renumber IDs); might be long

Surrogate Key (AutoNumber)

An arbitrary number generated by the system: 1, 2, 3, 4...

Pros: Never changes, always unique, minimal storage

Cons: Meaningless to users, requires a natural key for indexing

IGCSE Insight: For this course, natural keys (like CitizenID) are preferred because they demonstrate understanding of data modeling. Surrogate keys are acceptable but require justification.

Common Data Type Mistakes

❌ Storing Phone Numbers as Numbers →

Problem: Phone numbers can start with 0 (which Numbers ignore). 0123456789 becomes 123456789.

Solution: Store as Text.

❌ Using Currency for Quantities →

Problem: Currency formats add symbol and decimals. A quantity of 5 becomes "$5.00", which confuses calculations.

Solution: Use Number (Integer) for quantities, Currency only for money.

❌ Storing Dates as Text ('25/03/2050') →

Problem: Text dates can't be sorted chronologically. "01/12/2050" appears before "31/01/2050" alphabetically, but it's actually later.

Solution: Use Date/Time. The database handles format display; internally it stores as a number (days since reference point) for sorting.

Step-by-Step Build Instructions

📍 STEP 1: Create a New Database →
  1. Open MS Access
  2. Click Blank Desktop Database
  3. Name it: Medellin2050_SmartCity.accdb
  4. Choose your folder and click Create

✓ You now have an empty database.

📍 STEP 2: Switch to Design View →
  1. Access opens a default Table1
  2. Right-click the table tab and select Design View
  3. Save the table as CITIZENS when prompted

✓ You're now in Design View where you define fields.

📍 STEP 3: Define the Field Structure →

Enter these field definitions exactly:

Field Name Data Type Size (if applicable) Description

IsVerified Yes/No — Has identity been verified?
DistrictAssigned Short Text 30 District name

✓ All fields are now defined with the correct types.

📍 STEP 4: Set the Primary Key →
  1. Click on the CitizenID row
  2. Click the Primary Key button (🔑) in the Design ribbon
  3. A key icon will appear next to CitizenID

✓ CitizenID is now the fingerprint of the CITIZENS table. If you try to enter C-00101 twice, Access will reject it.

📍 STEP 5: Demonstration of a Data Type Error →

This is a high-impact teaching moment!

  1. Temporarily change EnergyCredits to Short Text
  2. Switch to Datasheet View
  3. Try entering "plenty" in the EnergyCredits field
  4. Switch the field back to Number
  5. Access will warn: "Some data may be lost. Do you want to continue?"

⚠️ Lesson: This is why we design the schema BEFORE entering data. The data type is the law. Choose it wrong, and you corrupt your system.

📍 STEP 6: Enter All 10 Records →
  1. Switch back to Datasheet View
  2. Enter the 10 citizen records from the table above
  3. Press Tab to move between fields
  4. Access will enforce your data types automatically
