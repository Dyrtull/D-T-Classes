Basic Level: The Six Types of Validation

Think of validation types as different kinds of guards at city gates. Each checks a different kind of threat.

1️⃣ Presence Check

Makes sure the field is not empty (null).

Access tool: Required = Yes
Example: CitizenID must not be blank — every citizen needs an ID

2️⃣ Range Check

Ensures the value is within acceptable limits (min and max).

Access tool: Validation Rule
Example: Energy Credits must be between 0 and 9999 (no negatives!)

3️⃣ Length Check

Ensures text is the correct number of characters.

Access tool: Field Size or Input Mask
Example: CitizenID must be exactly 7 characters (C-00101)

4️⃣ Format Check

Ensures the value follows a specific pattern or structure.

Access tool: Input Mask
Example: CitizenID must be Letter-Hyphen-5Digits (C-00101, not C00101 or CC-001)

5️⃣ Type Check

Ensures the value is the correct data type (automatic).

Access tool: Data Type field setting
Example: Age is Number — text like "twenty-five" is rejected automatically

6️⃣ Lookup Check (List Check)

Ensures the value is one of a predefined set of options.

Access tool: Lookup Wizard + Limit To List = Yes
Example: Sector must be one of: El Poblado, Laureles, Envigado, Belén, Centro (nothing else)

⚠️ Critical Understanding

Validation is NOT accuracy-checking. A validation rule checks that data is reasonable, complete, and within rules. But it cannot check if data is true or correct. A sensor could malfunction and report 45 when the real value is 92. Validation would accept 45 because it's in the valid range. The database cannot detect lies—only nonsense.

Intermediate Level: Implementing Validation

The Three Validation Mechanisms in MS Access

Access gives you three tools to enforce rules. Each serves a different purpose:

🔒 Mechanism 1: Required = Yes (Presence Check) →

What it does: Prevents the field from being left empty (null).

When to use: Any field that must have a value.

Location: Field Properties → Required

Example: CitizenID field: Required = Yes

⚠️ Note: This prevents NULL, but not blank spaces. For stronger checking, combine with Input Mask or Validation Rule.

📋 Mechanism 2: Input Mask (Format & Length Check) →

What it does: Enforces a specific pattern. Won't let you type anything that doesn't match.

Syntax:

L\-00000;0;_

Symbols:
L = one required letter
\- = literal hyphen (the backslash escapes it)
0 = one required digit
9 = one optional digit
_ = placeholder character shown while typing

Example: CitizenID with mask L\-00000 produces C-00101 format

⚙️ Mechanism 3: Validation Rule (Range, Logic Checks) →

What it does: Uses an expression to check logical conditions.

Common Examples:

>=0 And <=9999

Range check: value must be 0 to 9999

>=#01/01/2030# And <=Date()

Date range: between 2030 and today (Date() = today)

<> "" Or <> " "

Not empty and not just spaces

Key Operators:
= (equals) | <> (not equals) | < (less) |> (greater)
<= (less or equal) |>= (greater or equal)
And (both must be true) | Or (either can be true)

Why Validation Matters: The 2 AM Scenario Explained

Problem 1: Date of 99/99/9999
Solution: Range check on DateOfBirth
>=#01/01/1950# And <=Date()< /span> This accepts births between 1950 and today. 99/99/9999 is rejected immediately.
Problem 2: Energy Credits = -450000
Solution: Range check on EnergyCredits
>=0 And <=9999< /span> Negative numbers are impossible in the city's system. This rule rejects them.
Problem 3: CitizenID left blank
Solution: Required = Yes on CitizenID
The database won't let you skip to the next field. You must fill it in. Combined with Input Mask, it's bulletproof.

Advanced Level: Complex Rules & Edge Cases

Building Compound Validation Rules

Sometimes a single condition isn't enough. You need to combine multiple checks using And/Or logic.

Example 1: Age with Both Presence and Range >=18 And <=100

But combine with Required = Yes to ensure it's never null. This forces both the field to have a value AND that value to be reasonable.

Example 2: PollutionIndex (0-100) OR Status is "Inactive" (PollutionIndex >=0 And PollutionIndex <=100) Or Status="Inactive"

If the sensor is inactive, we don't validate the reading. Advanced logic that models real-world rules.

The Lookup Check Deep Dive

A common misconception: A dropdown list is not validation. It's just convenience. Limit To List = Yes is the actual validation that enforces it.

Limit To List = No: Dropdown shows suggestions, but you can type anything. Invalid!

Limit To List = Yes: Dropdown only. Cannot type anything else. Valid validation!

Validation Expressions: Advanced Syntax

Scenario Expression

🛠️ Guided Demonstration: Building CITIZENS Validation (0:25 – 0:55)

Open your Session 1 database: Medellin2050_SmartCity.accdb
Open CITIZENS table in Design View.
Follow along as we add validation to each field.

📍 FIELD 1: CitizenID (Presence + Format + Length) →
  1. Click the CitizenID row in Design View
  2. In Field Properties at the bottom:
    • Field Size: 7
    • Required: Yes
    • Input Mask: L\-00000;0;_
    • Validation Text: "Citizen ID must follow the format: one letter, a hyphen, and 5 digits (e.g., C-00101)"

✓ CitizenID is now bulletproof. It must exist, it must be 7 characters, and it must follow the exact pattern.

📍 FIELD 2: EnergyCredits (Range Check) →
  1. Click the EnergyCredits row
  2. In Field Properties:
    • Validation Rule: >=0 And <=9999< /li>
    • Validation Text: "Energy Credits must be a value between 0 and 9999."
  3. Try entering -50 in Datasheet View → Access blocks it with your custom message

✓ No negative energy credits allowed. The city's economy is protected.

📍 FIELD 3: DateOfBirth (Date Range Check) →
  1. Click DateOfBirth row
  2. In Field Properties:
    • Format: dd/mm/yyyy
    • Validation Rule: >=#01/01/1950# And <=Date()< /li>
    • Validation Text: "Date of Birth must be between 01/01/1950 and today."

✓ Impossible dates (like 99/99/9999) are rejected instantly. The Date() function makes this self-updating—no need to change it each year.

📍 FIELD 4: Sector (Lookup Check with Limit To List) →
  1. Click Sector row (assuming it exists; if not, add it as Short Text)
  2. CRITICAL: In Field Properties, set:
    • Limit To List: YES
    • Allow Value List Edits: NO

⚠️ Without "Limit To List = Yes", the dropdown is just a suggestion. With it, it becomes actual validation.

📍 FIELD 5: IsVerified (Automatic via Boolean) →

No extra validation needed! Yes/No fields cannot be null in Access—they default to No. That means this field is automatically presence-checked by its data type. Access handles it for you.

📍 FIELD 6: RegistrationDate (Self-Updating Date Range) →
  1. Click RegistrationDate row
  2. In Field Properties:
    • Validation Rule: >=#01/01/2030# And <=Date()< /li>
    • Validation Text: "Registration date must be between 01/01/2030 (city founding) and today."

✓ The Smart City didn't exist before 2030. This rule reflects institutional reality.
